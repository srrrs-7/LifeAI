# 💡 AI Context Hub

> Claude Codeとの対話ログを中央管理し、組織・チーム・プロジェクトに最適なコンテキストを継続的に自動生成するAIサービスです。

## 📋 概要

AI Context Hubは、Claude Codeとの対話ログを一元的に収集・分析し、そこから得られるinsightsをもとに、プロジェクトや組織に最適化されたコンテキストドキュメントを自動生成するサービスです。
チームメンバー全員が常に最新のコンテキストを共有できる仕組みを提供します。
これにより、AIとの対話品質をチーム全体で底上げし、属人化を排除します。

---

## 🌱 背景・きっかけ

プロジェクト立ち上げ段階では、AIに渡すコンテキストの蓄積が大きな課題になります。
プロジェクト開始後に蓄積されるコンテキストも、チーム全体で常に共有できる状態が理想です。
しかし現状では、個人単位・任意のタイミングでドキュメントを手動整備しており、属人的な運用に留まっています。

---

## 👥 対象ユーザー

AIを活用して開発・業務を行うすべてのビジネスパーソンが対象です。開発者に限りません。

- **メインターゲット**: AIを日常的に業務で活用しているチーム（エンジニア、PM、デザイナーなど職種を問わない）
- **利用シーン**: チーム開発でAIに繰り返し同じ説明をしている場面、メンバー間でAI出力品質に差が出ている場面

---

## 🔍 課題・ペイン

AIを活用するチームが直面する、コンテキスト共有に関する3つの課題です。

| 課題 | 深刻度 | 現状の対処 |
|------|--------|-----------|
| 毎回同じ説明をAIにしないといけない | 高 | 個人がその都度手動で説明を入力 |
| チームメンバーごとにAIの出力品質がバラバラ | 高 | 個人の経験・スキルに依存 |
| 自分が知らないコンテキストを人を介することでしかAIに伝えられない | 中 | 口頭やチャットで都度確認してからAIに入力 |

---

## 🛠️ 解決アプローチ

2つのアプローチで対話ログの収集とinsights生成を実現します。

### アプローチ1: Git連携（コードベース統合型）

1. Git等のコードベース上に対話ログを収集するhooksを作成する
2. 収集したログをもとにCI/CDパイプラインでinsightsを生成する
3. 生成されたinsightsをドキュメントに自動反映する（定期実行）

### アプローチ2: テレメトリ集約型

1. OpenTelemetry等の標準プロトコルを使用して対話ログを送信する
2. 特定のサーバに集約して一元管理する
3. 集約データからinsightsを生成し、コンテキストドキュメントを更新する

## 🔄 既存の代替手段

現時点で直接的な競合サービスは存在しないと考えています。

| 代替手段 | 長所 | 短所 |
|---------|------|------|
| 個人単位での手動ドキュメント整備 | 導入コストゼロ、個人の裁量で柔軟に運用可能 | 属人的で品質にバラつきが出る、更新タイミングが不定期、チーム共有が困難 |

---

## ✨ 差別化・独自性

- **導入の手軽さ**: 既存のGitワークフローやCI/CDパイプラインに組み込む形で、低コストに導入できます。新しいツールの学習コストを最小限に抑えます。
- **コンテキスト境界の制御**: 明確なコンテキスト境界を設けることで、情報漏洩やコンテキスト汚染を防止します。プロジェクト・チーム・組織の各レベルで適切なスコープを維持します。

---

## ⚠️ リスク・前提条件

- **リスク1（情報漏洩）**: ログデータの集約境界が明確でないと、PII（個人識別情報）などの機密情報をコンテキストに含めてしまう危険があります。コンテキスト境界の設計が不十分だと、意図しない情報が他チームに漏洩する可能性があります。
- **リスク2（プラットフォーム依存）**: Claudeを前提とした開発になるため、Anthropicのサービス変更やAPI仕様変更の影響を受けます。
- **前提条件**: チームがClaude Codeを日常的に利用していること。対話ログの収集に同意が得られること。
- **検証すべき仮説**: 対話ログから生成されたinsightsが、実際にAI出力品質の向上に寄与するか。

## 📐 実現可能性

### 既存基盤の活用

LifeAI プロジェクトの `insight-report` スキルが、本アイデアの技術基盤として活用できます。

| 要素 | 状態 | 詳細 |
|------|------|------|
| ログ収集 | **解決済み** | Claude Code が `~/.claude/projects/` に JSONL 形式で自動保存 |
| ログパース | **解決済み** | `analyze-logs.py` がトークン・ツール・エラー等を構造化解析 |
| 分析→改善フロー | **解決済み** | insight-report が「分析 → レポート → CLAUDE.md/settings 修正」のパターンを確立済み |
| 会話パターン分析 | **一部実装** | Agent C が JSONL を直接読み、リクエストパターン分類・応答品質評価を実施 |

### 残る技術的チャレンジ

現在の insight-report は**運用メトリクス**（トークン効率、ツール使用頻度、エラーパターン）を扱っています。AI Context Hub に必要なのは**意味的知識の抽出**です。

| 現在の insight-report | AI Context Hub に必要なもの |
|----------------------|---------------------------|
| トークン使用量の集計 | プロジェクトで学んだ技術的知見 |
| ツール使用頻度 | チームで合意されたコーディング規約 |
| エラーパターン検出 | アーキテクチャ上の意思決定の記録 |
| キャッシュ効率 | 「このライブラリはこう使う」等の暗黙知 |

ただし、Agent C の会話パターン分析を拡張し、セマンティックな insights 抽出に発展させるのは自然なステップです。抽出品質の向上はプロンプト設計の反復で改善可能な領域です。

### MVP ステップ

| フェーズ | 内容 | 見積 |
|---------|------|------|
| **Phase 0** | insight-report の Agent C を拡張し、対話ログから「プロジェクト知識・規約・決定事項」を抽出するプロンプトを追加 | 数日 |
| **Phase 1** | 抽出結果を CLAUDE.md に自動反映する仕組みを追加（既存の Step 3 修正フローを活用） | 数日 |
| **Phase 2** | Git hooks または CI/CD で定期実行化 | 1日 |
| **Phase 3** | 複数メンバーのログを集約する仕組み（ログを共有ディレクトリまたはリポジトリに同期） | 1〜2週間 |

Phase 0〜2 は既存コードベースの拡張だけで実現可能です。

## 🎯 ネクストアクション

1. **[高]** insight-report の Agent C 拡張 — 対話ログから意味的知識（技術知見・規約・意思決定）を抽出するプロンプトを設計・検証する
2. **[高]** CLAUDE.md 自動反映の仕組み構築 — 抽出した insights を既存の修正フローで CLAUDE.md に反映する
3. **[中]** 定期実行化 — Git hooks / CI/CD で insights 抽出・反映を自動化する
4. **[中]** チーム展開 — 複数メンバーのログ集約と、コンテキスト境界の設計・実装
5. **[低]** PII フィルタリング — ログ集約時の機密情報検出・除外の仕組みを追加
