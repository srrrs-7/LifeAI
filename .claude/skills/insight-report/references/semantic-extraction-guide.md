# Semantic Knowledge Extraction Guide

Agent D がセッションログから意味的知識を抽出する際のリファレンスです。

## 抽出カテゴリ

### 1. technical_insight（技術知見）

プロジェクト固有の技術的な発見や学び。

- **判定基準**: ライブラリの使い方、API の振る舞い、環境固有の注意点、パフォーマンス特性など、プロジェクトで実際に確認された技術的事実
- **具体例**:
  - 「このライブラリの X メソッドは Y の場合に Z の挙動になる」
  - 「本番環境では A の設定が必要」
  - 「B のバージョン X.Y 以降で C の仕様が変わった」
- **CLAUDE.md 反映先**: Architecture セクション、または新規 Technical Notes セクション

### 2. convention（規約・パターン）

チームで合意された、または繰り返し適用されているコーディング規約やパターン。

- **判定基準**: 命名規則、ディレクトリ構成、コードスタイル、エラーハンドリングの方針など
- **具体例**:
  - 「テストファイルは `*_test.rs` の命名で統一」
  - 「エラー型は `thiserror` を使用する」
  - 「API レスポンスは常に `Result<T, AppError>` で返す」
- **CLAUDE.md 反映先**: Coding Conventions セクション

### 3. decision（意思決定）

アーキテクチャや技術選定に関する意思決定とその理由。

- **判定基準**: 「なぜ X を選んだか」「なぜ Y ではなく Z にしたか」という判断の記録
- **具体例**:
  - 「状態管理に X を採用した理由は Y」
  - 「モノレポ構成にしたのは Z のため」
  - 「このモジュールを分離したのは A の懸念があったため」
- **CLAUDE.md 反映先**: Architecture セクション（判断理由付き）

### 4. workflow（ワークフロー）

繰り返し実行される手順やベストプラクティス。

- **判定基準**: 開発・デプロイ・テスト等で定型化された手順、推奨される作業順序
- **具体例**:
  - 「リリース前に X → Y → Z の順でチェックする」
  - 「新機能追加時は A を先に作成してから B に着手する」
  - 「環境構築は `make setup` → `make hooks` → `make check` の順」
- **CLAUDE.md 反映先**: Commands セクション、または新規 Workflows セクション

### 5. known_issue（既知の問題）

回避策付きの問題やハマりポイント。

- **判定基準**: 既知のバグ、制限事項、ワークアラウンドが確立された問題
- **具体例**:
  - 「X の場合に Y が発生するので Z で回避する」
  - 「A のバージョン B には C のバグがある」
  - 「テスト実行時に D が起きたら E を確認する」
- **CLAUDE.md 反映先**: 新規 Known Issues セクション

---

## Confidence 判定基準

| レベル | 基準 |
|--------|------|
| **high** | 複数のセッションで繰り返し言及されている。または、明示的な合意・決定として記録されている |
| **medium** | 2回以上言及されている。または、コードの修正を伴う具体的な議論に基づいている |
| **low** | 1回のみの言及。探索的な議論や仮説段階のもの |

---

## 除外パターン

以下に該当するものは抽出しない:

1. **一般的な知識**: プロジェクト固有でない一般的なプログラミング知識やベストプラクティス
   - 例: 「Rust では所有権を意識する」「テストを書くべき」
2. **一時的なデバッグ**: 問題解決のための試行錯誤で、最終的に不採用になったアプローチ
   - 例: 「X を試したがうまくいかなかった」（ただし、回避策が確立された場合は known_issue として抽出）
3. **プロジェクト外の話題**: 雑談、一般的な質問、他プロジェクトに関する議論
4. **ツール操作の指示**: Claude Code のツール使用に関する指示やフィードバック
   - 例: 「Read ツールを使って」「Bash ではなく Grep を使って」（ただし、これが CLAUDE.md Instructions に反映すべき内容なら convention として抽出可）
5. **既に CLAUDE.md に記載済みの内容**: 現在の CLAUDE.md と意味的に重複する情報

---

## 出力フォーマット

各抽出アイテムは以下の構造で報告する:

```
- **カテゴリ**: technical_insight | convention | decision | workflow | known_issue
- **要約**: 1行の要約（これだけで内容が伝わるように）
- **詳細**: 具体的な内容（2-3行。根拠となる事実を含む）
- **ソースセッション**: セッションID
- **信頼度**: high | medium | low
- **反映先**: CLAUDE.md の対象セクション名
```

---

## 注意事項

- ユーザーの発言をできるだけ活かし、意味を変えない範囲で簡潔にまとめる
- 同じ知識が複数セッションで出現する場合は統合し、confidence を上げる
- PII（個人名、メールアドレス、API キー等）が含まれる場合は除外する
- 判断に迷う場合は confidence: low で抽出し、ユーザーの判断に委ねる
